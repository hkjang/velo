# Tomcat 10 대비 velo가 더 빠르게 느껴지는 이유

## 요약

체감상 `velo`가 `Tomcat 10`보다 더 빠르게 보인다고 해서, 그 이유를 `Java 21` 하나로만 설명하기는 어렵다.

실제로는 다음 세 가지가 함께 작용했을 가능성이 크다.

1. `Java 21` 자체의 개선 효과
2. `velo`의 더 짧고 단순한 기동/배포 경로
3. `Tomcat 10`보다 아직 더 좁은 기본 기능 범위

결론부터 말하면, **Java 21도 분명 영향이 있지만, 주된 이유는 `velo`가 현재 더 가볍고 덜 복합적인 컨테이너이기 때문일 가능성이 높다.**

## Java 21만의 효과인가?

부분적으로는 맞다.

`Java 21`로 올라가면 일반적으로 다음이 좋아질 수 있다.

- 클래스 로딩 및 런타임 라이브러리 최적화
- GC 및 메모리 관리의 개선
- TLS, NIO, 스레드 스케줄링 계층의 개선
- 전체적으로 더 최신 JIT 최적화

하지만 이번 `velo` 코드 기준으로 보면, **동기 서블릿 요청 처리 경로 자체가 virtual thread 중심으로 돌아가지는 않는다.**

`SimpleServletContainer`는 비동기 처리용 executor에서만 virtual thread를 사용한다.

```java
private final ScheduledExecutorService asyncExecutor =
        Executors.newScheduledThreadPool(Runtime.getRuntime().availableProcessors(),
                Thread.ofVirtual().name("velo-async-", 0).factory());
```

즉, 일반적인 sync request/response가 빨라 보이는 현상을 모두 virtual thread나 Java 21 덕분으로 보기는 어렵다.

**정리하면 Java 21은 플러스 요인이지만, 단독 원인으로 보기는 어렵다.**

## velo가 더 빠르게 느껴질 수 있는 구조적 이유

### 1. 서버 기동 경로가 짧다

`velo`의 메인 진입점은 비교적 단순하다.

- 설정 로드
- `SimpleServletContainer` 생성
- 관리용 앱과 샘플 앱 등록
- `deploy/` 디렉토리의 WAR 목록 순회
- `WarDeployer`로 앱 로드
- Netty 서버 시작

즉, 현재 구조는 "필요한 것만 올리고 바로 listen"에 가깝다.

반면 Tomcat 10은 범용 서블릿 컨테이너이기 때문에 보통 다음과 같은 더 넓은 초기화 책임을 가진다.

- Connector / Service / Engine / Host / Context 계층 초기화
- 더 넓은 호환성 범위를 위한 메타데이터 처리
- 기본 파이프라인/밸브/리소스 계층 초기화
- 더 많은 운영/호환성 훅 유지

이 차이만으로도 서버 시작과 앱 attach 시점이 더 무겁게 느껴질 수 있다.

### 2. WAR 배포 과정이 단순하다

현재 `velo`의 WAR 배포는 대체로 다음 순서다.

1. WAR 압축 해제
2. `WEB-INF/web.xml` 파싱
3. 앱 전용 classloader 생성
4. 서블릿/필터/리스너 인스턴스화
5. 필요하면 annotation scan 수행
6. 기본 정적 리소스 서블릿 등록
7. `SimpleServletApplication` 빌드 후 컨테이너에 탑재

즉, 지금 배포 흐름은 비교적 직선적이다.

Tomcat 10은 보통 더 많은 범용 처리와 호환성 계층을 감당한다. 그래서 동일한 WAR라도:

- 더 많은 메타데이터를 읽고
- 더 많은 초기화 훅을 고려하고
- 더 많은 표준/확장 포인트를 유지

해야 할 가능성이 크다.

### 3. 요청 처리 경로도 더 짧다

현재 `velo`는 `HttpHandlerRegistry`에서 `servletContainer::handle`로 바로 떨어지는 구조이고, 내부도 상당히 단순한 편이다.

대략 흐름은 다음과 같다.

1. Netty가 요청 수신
2. `HttpHandlerRegistry`가 라우팅
3. `SimpleServletContainer`가 context path와 servlet path를 매칭
4. 필터 체인 계산
5. 서블릿 호출

Tomcat 10은 범용 컨테이너답게 내부 계층과 확장 지점이 더 많다. 이 차이는 평균 응답시간보다도 **첫 요청 체감**과 **정적/간단 요청의 가벼움**에서 더 크게 느껴질 수 있다.

### 4. 정적 리소스 서빙 경로가 가볍다

`velo`의 `StaticResourceServlet`은 다음에 집중한다.

- 단순 MIME 해석
- 캐시 헤더
- 조건부 GET
- precompressed asset 대응
- SPA fallback

즉, 최신 브라우저용 정적 리소스 서빙에 맞춘 비교적 직접적인 경로다.

특히 HTML/JS/CSS 파일을 많이 주고받는 브라우저형 앱에서는 이 경로가 가볍게 느껴질 수 있다.

## Tomcat보다 빨라 보일 수 있지만, 그만큼 덜 하고 있을 수도 있다

이 부분이 중요하다.

현재 `velo`는 README와 `/info` 응답에서 스스로도 `minimal`한 서블릿 호환 범위를 전제로 둔다.

즉, **Tomcat 10보다 기능이 더 적게 구현되어 있어서 빠른 부분도 있다.**

예를 들면 현재 코드베이스에서는 다음을 확인할 수 있다.

- `ServletContainerInitializer` 지원 흔적이 없다
- `web-fragment` 처리 흔적이 없다
- Tomcat식 광범위한 범용 컨테이너 인프라를 그대로 재현하지 않는다
- JSP 지원은 존재하지만 여전히 실험적이고 단순화된 구현이다

이 말은 곧:

- `velo`가 더 빠르게 보일 수는 있지만
- 그것이 곧 Tomcat 10보다 "더 최적화된 범용 컨테이너"라는 뜻은 아니고
- **현재는 "더 적은 일을 하기 때문에 빠른 부분"도 분명 있다**

는 의미다.

## 어떤 경우에 특히 velo가 더 빨라 보일 수 있나

### 1. 앱 시작/재배포

다음 조건이면 `velo`가 더 빨라 보일 가능성이 높다.

- WAR가 복잡하지 않다
- `web.xml`과 annotation 수가 많지 않다
- JSP/TLD/태그 라이브러리 의존성이 무겁지 않다
- Tomcat 쪽에서 더 넓은 초기화 작업을 수행한다

### 2. 정적 UI 로딩

다음 조건이면 초기 화면이 더 빨라 보일 수 있다.

- HTML/JS/CSS 요청이 많다
- 필터 체인이 짧다
- 정적 리소스 응답이 대부분이다
- keep-alive 기반으로 가볍게 여러 요청이 이어진다

### 3. 단순 서블릿 응답

다음 같은 요청은 `velo`가 유리하게 보일 수 있다.

- JSON 반환
- 단순 세션 조회
- 필터 수가 적은 CRUD 요청

이런 케이스는 컨테이너가 범용 처리 대신 단순 dispatch만 해도 되기 때문이다.

## 반대로 velo가 Tomcat보다 느려질 수도 있는 경우

다음 상황에서는 Tomcat 10이 오히려 더 안정적이거나 빠를 수 있다.

- 복잡한 JSP/JSTL/TLD 사용량이 많은 앱
- 표준 호환 포인트가 많은 레거시 앱
- 장시간 운영에서 검증된 튜닝 포인트가 필요한 경우
- 복잡한 클래스패스/라이브러리 조합이 많은 경우
- 컨테이너 내부 캐시, 스캐닝, 최적화가 오랫동안 다듬어진 시나리오

특히 Tomcat은 오랜 기간 다양한 워크로드에서 다듬어진 범용 컨테이너다. 따라서 단순 체감이 아니라 장기 부하, 예외 상황, 희귀 호환성까지 포함하면 평가가 달라질 수 있다.

## 이번 관찰에 대한 현실적인 해석

현재 repo 기준으로 가장 현실적인 해석은 아래와 같다.

### 가능성이 큰 순서

1. `velo`가 더 가벼운 컨테이너라서 startup/deploy/request path가 짧다
2. `Java 21`이 추가로 플러스 효과를 준다
3. Tomcat 10이 더 넓은 범용 초기화/호환성 작업을 수행한다

즉, **"Java 21이라서 빠르다" 보다는 "Java 21 + 더 단순한 컨테이너 구조"가 더 맞는 설명**이다.

## 공정하게 비교하는 방법

정확히 확인하려면 다음처럼 비교하는 것이 좋다.

1. Tomcat 10과 `velo`를 **같은 JDK 21**로 맞춘다.
2. 같은 서버, 같은 WAR, 같은 OS 조건에서 측정한다.
3. 비교 항목을 분리한다.
   - 프로세스 시작 시간
   - WAR deploy 완료 시간
   - 첫 요청 응답 시간
   - 정적 리소스 응답 시간
   - 일반 서블릿 응답 시간
4. 각 항목을 최소 5회 이상 반복 측정한다.
5. 관리 앱, hot deploy, 디버그 로그처럼 비교에 불필요한 옵션은 양쪽 모두 최대한 맞춘다.

## 권장 결론

팀 내 설명용으로는 아래 문장을 추천한다.

> `velo`가 Tomcat 10보다 빠르게 느껴지는 것은 Java 21 영향도 있지만, 더 큰 이유는 현재 `velo`가 더 가볍고 단순한 컨테이너 구조를 가지고 있기 때문이다. 다만 이 차이는 일부 기능을 덜 수행하거나 더 좁은 호환 범위를 전제로 한 결과일 수도 있으므로, 동일 JDK와 동일 조건에서 startup, deploy, request latency를 나눠 다시 측정하는 것이 맞다.
